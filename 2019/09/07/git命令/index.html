git命令的使用

Posted on 2019-09-07 | In 命令、语法 | | Visitors:

git init:在当前目录新建一个Git代码库

git clone [url]：将项目和完整代码克隆到当前所在工作目录

git add [文件名]：将对应的文件上传到暂存区

git commit -m [提交文件的描述信息]：将暂存区的文件上传到仓库

git push origin dev:该命令是将本地dev分支推送到远程仓库对应的远程仓库。
git remote：查看远程仓库的信息。

注：当你从远程仓库克隆时，实际上Git自动把本地的master分支和远程的master分支对应起来了，并且，远程仓库的默认名称是origin。

git status：查看变更的文件详细信息一般用来辅助查看git add和git commit -m[message]所对应的上传信息是否成功。

注：一般可以将对应的github仓库克隆到本地，然后将要上传到github的文件添加到该克隆文件夹下，该文件夹下还会有一个隐藏的.git文件，然后使用git add . 命令可以将本地文件夹下的所有文件上传到该github远程仓库。其中.类似与linux命令下.含义相同代表着当前目录，因此 git add .的含义是将当前目录下的所有文件添加到暂存区。

git diff [文件名]:如果你修改了一个文件几天之后你忘记了修改内容，那么可以使用git diff 命令来查看上一次修改情况。

git log:显示从最近到最远的提交日志，如果日志较多可以采用git log –pretty=oneline,出现的一串串数字是git的commit id

git reset –hard HEAD^:其中HEAD表示当前版本，整个命令表示把当前版本回退到上一个版本，HEAD^表示上一个版本，HEAD^^表示上两个版本，如果回退到20个版本可以使用HEAD~20回退到上一个版本之后，这时候使用git long你会发现原来最新的版本没有了，这个时候如果你后悔了，想回到原来的最新版本，此时如果你的命令窗口没有关闭你可以通过往前翻命令找到原来最新版本的commit id使用git trset –hard [commit ID]回到原来最新的版本。如果你关闭了命令窗口，可以通过使用git reflog命令来查看你原先的命令记录历史从而找到 commit ID回退到最新版本

git checkout –[文件名]：把文件在工作区的修改全部撤销，分成两种情况：一种是文件从修改之后还没有还没有被放到暂存区，这时候撤销修改就是回退到原先未修改的状态。一种是文件已经添加到暂存区，又作了修改，这个时候撤销修改就是回到添加到暂存区时的那个最初状态。git reset HEAD[文件名]可以把暂存区的修改撤销掉重新放回工作区。

git rm [文件名]：删除文件这个时候如果使用git status会告诉你文件被删除了，如果确认删除则使用git commit来提交即可。如果误删除了，可以使用git checkout –[文件名]既可以恢复原先删除的文件。该操作实际上是使用远程仓库里面的文件替换工作区的文件。

git branch:查看分支状态。

git branch [名称]：创建一个分支

git checkout -b [名称]或者是git switch -c[名称]：切换分支

注：git switch不适合旧的git版本。

git checkout -b[名称]或者git switch -c[名称]:创建+切换分支的命令集合。

git merge [名称]：合并某分支到当前分支。

git branch -d[名称]：删除某分支。

注:合并分支适合于我们准备开发一个项目代码没写完提交的话会影响别人的工作，不提交害怕代码丢失，这个时候就可以使用分支管理，创建一个属于自己的分支，你在自己的分支上干活，别人看不到，想提交提交，而且别人还可以在你原先的分支上继续工作，当你完成项目的时候，一次性合并到原先的分支即可。分支合并有时候会出现冲突，这个时候我们首先要先解决冲突在提交。

git stash:将当前工作现场储藏其来，等待以后恢复现场继续工作。

git stash list：查看储藏起来的工作现场。

git stash apply:恢复储藏的工作现场，但是这个时候你在git stash会发现原先储藏的那个工作现场并没有被删除，如果使用git stash drop:就可以在恢复工作现场的同时删除stash的内容。

注该命令一般适合场景为你正在做项目，老板让你在两个小时内修改一个文件的bug，你当前项目还要一段时间完成这个时候你就可以先保存当前的工作场景，在完成老板的任务之后恢复工作场景继续做自己的项目。

git cherry-pick[文件名或者commit ID]：复制特定的一个提交到当前分支。

注：如果你刚才在别的分支修改了代码现在要求重新在另一个分支修改一遍，笨拙的做法是再重新像刚才那样执行一次修改，简便的方法是使用git cherry-pick[文件名或者commit ID]就可以将在别的分支上修改好的bug文件复制到新的分支。一般使用commit ID。
